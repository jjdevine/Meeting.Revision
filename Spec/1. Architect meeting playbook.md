## Architect meeting playbook

Use this pattern in the room:

> This is probably a **[rules / workflow / eventing / integration / identity / reporting]** problem.
> I see a **simple option**, a **platform option**, and a **heavyweight option**.
> The choice depends on **[3-4 criteria]**.
> My default is **[X]** unless we need **[Y]**.

That is the whole game. You do **not** need the final answer on the spot. You need a better first frame than everyone else.

---

## 1) Rules / decisioning

Think: leave eligibility, approval thresholds, payroll exceptions, benefits rules.

**Ask first**

- Who changes the rules?
- How often do they change?
- Do you need audit history or replay by rule version?
- Do non-developers need to author them?
- Are rules reused across multiple services?
- Do you need simulation/testing before publish?

**Default**

- Keep developer-owned rules in code plus versioned config first.
- Azure App Configuration is for centralized settings and feature flags, and it supports immutable snapshots plus point-in-time revisions. Azure also now has a Logic Apps Rules Engine for **Standard** workflows, with Rules Composer for authoring and versioning rulesets. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-app-configuration/?utm_source=chatgpt.com "Azure App Configuration documentation | Microsoft Learn"))

**Bias for payroll**

- Core payroll calculation logic should usually stay code-first, with externalized tables/config where needed.
- Move to a real rules platform only when business ownership, auditability, frequent change, or shared reuse actually justify it.

**Say**

> Let’s separate config, decision tables, and a full rules platform. We should only move up the stack if the business needs ownership, versioning, replay, or reuse.

**Avoid**

- Using a rules engine to cover bad domain boundaries.
- Centralizing every policy into one giant shared “rules service.”
- Putting correctness-critical logic somewhere that is hard to test, version, and replay.

## 2) Workflow / orchestration

Think: onboarding/offboarding, approval chains, payroll exception handling, scheduled business processes.

**Ask first**

- Is this mostly system integration, or mostly domain logic?
- Are there human approvals or external SaaS steps?
- Is it long-running?
- Does it need retries, timeouts, compensation, or state tracking?
- Who owns the flow: app team or integration team?

**Default**

- Logic Apps is for automated workflows and enterprise integrations across cloud and on-prem systems.
- Durable Functions is for code-defined **stateful** workflows; the runtime manages state, checkpoints, retries, and long-running execution, and orchestrator code must be deterministic. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/logic-apps/logic-apps-overview?utm_source=chatgpt.com "Overview - Azure Logic Apps | Microsoft Learn"))

**Bias**

- **Connector / human / B2B heavy** -> Logic Apps.
- **Code-heavy domain saga** -> Durable Functions.
- **Simple synchronous service chain** -> do not invent orchestration.

**Say**

> We should decide whether this is integration-heavy or domain-logic-heavy. Integration-heavy points to Logic Apps. Code-owned long-running orchestration points to Durable.

**Avoid**

- Turning every multi-step process into orchestration.
- Putting core domain behavior into a connector flow just because it is visual.
- Building distributed transactions by wishful thinking.

## 3) Eventing / messaging

Think: `EmployeeCreated`, `EmploymentStatusChanged`, `PayrollRunCompleted`, audit streams, notifications.

**Ask first**

- Is this a **command**, an **event**, or a **stream**?
- One consumer or many?
- Do you need durable processing and retries?
- Is this business data or telemetry?
- Do consumers need independent scaling?

**Default**

- Azure Service Bus is the enterprise broker with queues and pub/sub topics.
- Azure Event Grid is the managed pub/sub event router for near-real-time event distribution.
- Azure Event Hubs is the high-throughput streaming/event-ingestion service. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/service-bus-messaging/compare-messaging-services?utm_source=chatgpt.com "Compare Azure messaging services - Azure Service Bus"))

**Bias**

- **Command to one processor** -> Service Bus.
- **Business notification to many consumers** -> Event Grid, unless you need broker semantics that pull you back to Service Bus topics.
- **Telemetry / audit firehose / clickstream** -> Event Hubs.

**Say**

> Before we pick a service, we need to decide whether we are sending a command, broadcasting an event, or ingesting a stream. Those are different problems.

**Avoid**

- Calling everything “event-driven.”
- Using Event Hubs for business workflows.
- Using a queue when what you really need is fan-out.
- Pretending consumers can be non-idempotent.

## 4) Integration / API / B2B

Think: HRIS, payroll providers, finance, identity, document signing, partner file exchange.

**Ask first**

- Is this API governance, protocol mediation, or process integration?
- Is the consumer internal, external, or partner?
- Is this synchronous or asynchronous?
- Are we dealing with SaaS connectors, files, or B2B formats?
- Do we need throttling, auth, transformation, or developer onboarding?

**Default**

- API Management policies cover auth, rate limiting, caching, and transformation, and APIM supports managed and self-hosted gateways.
- Logic Apps integration accounts handle B2B artifacts such as trading partners, agreements, maps, schemas, and certificates. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/api-management/api-management-howto-policies?utm_source=chatgpt.com "Policies in Azure API Management | Microsoft Learn"))

**Bias**

- **External API edge and governance** -> APIM.
- **Connector-heavy SaaS / file / B2B integration** -> Logic Apps.
- **Internal async decoupling** -> messaging first, not API-first.
- **Internal service-to-service** -> do not insert APIM everywhere unless there is a real governance reason.

**Say**

> This is either an API governance problem, a mediation problem, or an integration-flow problem. We should not collapse those into one product decision.

**Avoid**

- Treating APIM as the answer to every integration problem.
- Point-to-point sprawl.
- Synchronous integration for work that should be buffered.

## 5) Identity / access

Think: employee sign-in, manager access, HRBP privileges, payroll admin, service-to-service auth.

**Ask first**

- Is this workforce authentication, external consumer access, or workload identity?
- Is access user-to-app, app-to-app, or delegated on behalf of a user?
- What is identity, and what is business authorization?
- Do we need JML, privileged access, audit, separation of duties?

**Default**

- Microsoft Entra ID is the cloud identity directory/service, and the Microsoft identity platform supports app authentication and authorization with standards such as OAuth 2.0 and OIDC.
- Managed identities are the recommended option for Azure resources that support them, and APIM can use managed identity when calling backends. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/identity/?utm_source=chatgpt.com "Microsoft Entra ID documentation - Microsoft Entra ID"))

**Bias**

- **Workforce SSO / token issuance** -> Entra.
- **Azure service-to-service** -> managed identities first.
- **Business entitlements** -> keep in domain logic and data, not only in IdP groups.

**Say**

> Identity proves who the caller is. It does not decide payroll entitlements. We still need domain authorization.

**Avoid**

- Encoding all business authorization in directory groups.
- Hard-coding secrets when managed identity is available.
- Mixing identity roles with business roles without a model.

## 6) Reporting / analytics

Think: headcount, attrition, absence, comp reporting, payroll reconciliation, regulatory reporting.

**Ask first**

- Is this operational reporting or cross-domain analytics?
- Does it need fresh transactional data or periodic curated data?
- Who needs row-level restrictions?
- Is the report tied to one service or multiple bounded contexts?

**Default**

- Power BI uses semantic models and supports row-level security.
- In workspaces, RLS applies to **Viewers**; Admin/Member/Contributor roles have edit permissions, so RLS does not apply to them in the same way. ([Microsoft Learn](https://learn.microsoft.com/en-us/power-bi/connect-data/service-datasets-understand?utm_source=chatgpt.com "Semantic Models in the Power BI Service - Power BI | Microsoft Learn"))

**Bias**

- **Operational reporting** -> service-owned read models or replicas.
- **Cross-domain analytics** -> curated analytical store + semantic model + RLS.
- Do **not** build enterprise reporting by joining live microservice stores in place.

**Say**

> We should split operational reads from enterprise analytics. One is a service design question; the other is a data product question.

**Avoid**

- Reporting straight off live OLTP tables across services.
- Cross-service joins hidden inside app queries.
- Assuming RLS design is an afterthought.

## Lines worth memorizing

Use these until they become automatic:

- “This looks like a capability question before it is a product question.”
- “I see a simple option, a platform option, and a heavyweight option.”
- “The decision hinges on volatility, ownership, audit, latency, and reuse.”
- “Let’s separate authentication from business authorization.”
- “We need to decide whether this is a command, an event, or a stream.”
- “We should not jump to vendors until we know where on the complexity curve this sits.”

## Default stance for your kind of platform

For an Azure HR/payroll/talent platform, a sane default bias is:

- **Payroll calculations / critical policy logic** -> code + versioned config first; move to Rules Engine only when business ownership and audited change velocity really demand it. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-app-configuration/?utm_source=chatgpt.com "Azure App Configuration documentation | Microsoft Learn"))
- **Approvals / onboarding / connector-heavy business flows** -> Logic Apps. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/logic-apps/logic-apps-overview?utm_source=chatgpt.com "Overview - Azure Logic Apps | Microsoft Learn"))
- **Long-running code-owned sagas inside a bounded context** -> Durable Functions. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-functions/durable-functions/durable-functions-overview?utm_source=chatgpt.com "Durable Functions overview - Azure Durable | Microsoft Learn"))
- **Internal commands and durable async work** -> Service Bus. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/service-bus-messaging/service-bus-messaging-overview?utm_source=chatgpt.com "Introduction to Azure Service Bus Messaging - Azure Service Bus"))
- **Domain notifications / reactive fan-out** -> Event Grid. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/event-grid/?utm_source=chatgpt.com "Azure Event Grid documentation | Microsoft Learn"))
- **Telemetry / audit stream ingestion** -> Event Hubs. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/event-hubs/event-hubs-about?utm_source=chatgpt.com "What is Azure Event Hubs - Real-time data streaming platform - Azure ..."))
- **External API governance** -> APIM. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/api-management/api-management-howto-policies?utm_source=chatgpt.com "Policies in Azure API Management | Microsoft Learn"))
- **Workforce and workload identity** -> Entra + managed identities. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/identity/?utm_source=chatgpt.com "Microsoft Entra ID documentation - Microsoft Entra ID"))
- **Enterprise reporting** -> curated model + Power BI semantic model + RLS. ([Microsoft Learn](https://learn.microsoft.com/en-us/power-bi/connect-data/service-datasets-understand?utm_source=chatgpt.com "Semantic Models in the Power BI Service - Power BI | Microsoft Learn"))

Memorize the pattern, not the products. Products change. The framing does not.
