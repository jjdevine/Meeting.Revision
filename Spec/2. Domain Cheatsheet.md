## One-page cheat sheet — Azure HR / payroll / talent

**Use this opener in meetings**

> “This is mainly a **[rules / workflow / integration / identity / reporting]** decision.
> Default is the **simple option**. We only move to the **platform option** if **volatility, ownership, audit, latency, or reuse** justify it.”

### Payroll

- **Default:** Keep payroll calculation logic **code-first in the owning service**, with centralized config/flags in **App Configuration** and **Service Bus** for durable async processing. Only consider a separate rules platform when rule changes are frequent, business-owned, auditable by version, or reused across services. App Configuration is designed to centralize application settings, Azure Logic Apps Rules Engine is a low-code rules capability for **Standard Logic Apps workflows**, and Service Bus is positioned for high-value enterprise messaging such as financial transactions. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-app-configuration/ "Azure App Configuration documentation | Microsoft Learn"))
- **Ask:** Who changes the rule? How often? Do we need replay by rule version? Is this shared across services? Is simulation/testing required before publish?
- **Say:** “Payroll is correctness-critical. I’d stay code-first with versioned config unless we need business-authored, auditable, reusable rules.”
- **Avoid:** Giant shared rules services, non-versioned rate tables, no replay/audit story.

### Absence / Leave

- **Default:** Keep entitlement policy **service-owned and simple**. Use **Logic Apps** for approval-heavy or connector-heavy flows. Move to **Durable Functions** only when the process is code-centric, stateful, and long-running with retries or compensation. Logic Apps is the low-code workflow/integration platform; Durable Functions is the code-first stateful workflow option with managed checkpoints, retries, and recovery. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/logic-apps/logic-apps-overview "Overview - Azure Logic Apps | Microsoft Learn"))
- **Ask:** Straight-through or approval? Any SLA/timeouts? Calendar/public holiday rules? What happens on cancellation or retroactive change?
- **Say:** “This looks like policy plus workflow. I’d keep policy in the service and only externalize the process if the approvals and integrations justify it.”
- **Avoid:** Hiding core policy logic inside visual flows.

### Onboarding / Offboarding

- **Default:** Treat this as **orchestration plus identity lifecycle**. Put workforce identity and app access in **Microsoft Entra ID**, use **Logic Apps** where the work is mostly provisioning/integration across systems, use **Durable Functions** where the team owns a code-defined saga, and use **managed identities** for service-to-service calls instead of secrets. Entra ID manages user identities and access, and managed identities remove the need for applications to handle credentials directly. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/identity/ "Microsoft Entra ID documentation - Microsoft Entra ID | Microsoft Learn"))
- **Ask:** Is this mostly provisioning across SaaS/on-prem? Do we need waits, retries, compensation, or manual checkpoints? Who owns the failure recovery?
- **Say:** “This is orchestration, not just workflow. Connector-heavy points to Logic Apps; code-owned saga behavior points to Durable.”
- **Avoid:** Synchronous chains across many systems, secret sprawl, unclear rollback ownership.

### Benefits

- **Default:** Treat benefits as **eligibility rules plus partner integration**. Start with service-owned rules/config, use **Service Bus** when enrollment work must be processed reliably, use **API Management** at carrier/vendor API edges for auth, throttling, caching, and transformation, and use **Logic Apps** when the integration is connector-heavy or B2B-like. APIM policies are built for those edge concerns, and Logic Apps is the workflow/integration platform. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/service-bus-messaging/compare-messaging-services "Compare Azure messaging services - Azure Service Bus | Microsoft Learn"))
- **Ask:** How many carriers/vendors? API or file-based? Real-time or batch? Enrollment windows? Retro changes? SLA and reconciliation needs?
- **Say:** “This is rules plus partner integration. I wouldn’t jump to a benefits platform component unless partner complexity or business-owned rule churn actually forces it.”
- **Avoid:** Product-first thinking because “benefits are complex.”

### Identity / Access

- **Default:** Split **authentication** from **business authorization**. Use the **Microsoft identity platform / Entra** for workforce SSO and OAuth2/OIDC-based API access, use **managed identities** for Azure workloads, and keep payroll/HR entitlements in domain data instead of directory groups alone. The Microsoft identity platform supports OAuth 2.0 and OpenID Connect, and APIM can authenticate to backends by using managed identity tokens from Entra. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/identity-platform/v2-overview "Microsoft identity platform overview - Microsoft identity platform | Microsoft Learn"))
- **Ask:** Workforce or external? User-to-app or app-to-app? Business role or IT role? Any separation-of-duties or privileged access concerns?
- **Say:** “Identity proves who the caller is. It does not decide payroll entitlement. We still need domain authorization.”
- **Avoid:** Encoding payroll access purely in Entra groups.

### Reporting / Analytics

- **Default:** Separate **operational reads** from **enterprise analytics**. For service screens, use service-owned read models. For cross-domain HR/payroll/talent analysis, publish curated **Power BI semantic models** and enforce **RLS** where needed; in Power BI workspaces, RLS applies to **Viewer** users, while **Admin/Member/Contributor** roles have edit access and are not restricted the same way. ([Microsoft Learn](https://learn.microsoft.com/en-us/power-bi/connect-data/service-datasets-understand "Semantic Models in the Power BI Service - Power BI | Microsoft Learn"))
- **Ask:** Operational or analytical? Single service or cross-domain? Freshness requirement? PII? Row-level restrictions by manager, HRBP, country, or legal entity?
- **Say:** “We should not solve enterprise reporting by joining live microservice stores. That’s a separate analytics design.”
- **Avoid:** Cross-service joins over OLTP databases, undefined ownership of analytic definitions.

## Fast service map

- **Config / flags:** App Configuration. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-app-configuration/ "Azure App Configuration documentation | Microsoft Learn"))
- **Connector-heavy or human workflow:** Logic Apps. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/logic-apps/logic-apps-overview "Overview - Azure Logic Apps | Microsoft Learn"))
- **Code-owned long-running saga:** Durable Functions. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-functions/durable-functions/durable-functions-overview "Durable Functions overview - Azure Durable | Microsoft Learn"))
- **High-value business messaging:** Service Bus. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/service-bus-messaging/compare-messaging-services "Compare Azure messaging services - Azure Service Bus | Microsoft Learn"))
- **Reactive event fan-out:** Event Grid. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/service-bus-messaging/compare-messaging-services "Compare Azure messaging services - Azure Service Bus | Microsoft Learn"))
- **Telemetry / stream ingestion:** Event Hubs. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/service-bus-messaging/compare-messaging-services "Compare Azure messaging services - Azure Service Bus | Microsoft Learn"))
- **API edge governance:** API Management. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/api-management/api-management-howto-policies "Policies in Azure API Management | Microsoft Learn"))
- **Workforce identity:** Entra. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/identity/ "Microsoft Entra ID documentation - Microsoft Entra ID | Microsoft Learn"))
- **Analytics model:** Power BI semantic model. ([Microsoft Learn](https://learn.microsoft.com/en-us/power-bi/connect-data/service-datasets-understand "Semantic Models in the Power BI Service - Power BI | Microsoft Learn"))

## When someone jumps straight to a vendor

Use one of these:

- “What requirement forces us off the simple option?”
- “Who owns change, and how is that change versioned and audited?”
- “Is this a workflow problem or are we jumping to a product category too early?”
- “Are we solving one bounded context, or accidentally inventing a platform?”
